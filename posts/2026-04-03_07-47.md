## 二、核心特性深度解析：开发者必须掌握的 7 把利器

### 2.1 模块化系统 (Project Jigsaw)：终结“类路径地狱”

**引入版本**：JDK 9 | **关键词**：`module-info.java`, 强封装

还记得我开头那个崩溃的凌晨吗？罪魁祸首就是“类路径地狱”。在 JDK 9 之前，整个 JRE 是一个巨大的、黑盒式的单体，所有类都暴露在公共视野下，依赖冲突如同无解的死结。

**我踩过的坑：**
刚升级 JDK 9 时，我的老项目直接炸了。报错提示某些内部 API 无法访问。我当时气得想回退版本，后来才明白，这是 Java 在强行给我们“断奶”。

**破局之道：**
模块化系统通过 `module-info.java` 强制你显式声明依赖。这就好比给每个房间装了门禁，只有持有钥匙（明确声明）的人才能进入。

```java
// module-info.java
module com.myapp.core {
    // 显式声明我只需要这两个模块
    requires java.sql;
    requires com.fasterxml.jackson.databind;
    
    // 只暴露这个包给外部，其他内部实现全部隐藏
    exports com.myapp.api;
}
```

**反直觉的发现：**
起初我觉得这是束缚，直到我第一次使用 `jlink` 构建运行时镜像。原本 200MB+ 的 JRE，被裁剪成了只剩 30MB 的精简版，容器启动速度瞬间提升了 40%。**原来，所谓的“限制”，其实是通往轻量级的捷径。**

### 2.2 局部变量类型推断 (var)：在简洁与可读性之间找到平衡

**引入版本**：JDK 10/11 | **关键词**：`var`, 编译器推断

很多人一听到 `var` 就惊呼："Java 要变成动态类型语言了吗？”
**大错特错。** `var` 只是编译期的语法糖，类型安全底线纹丝不动。

**我踩过的坑：**
刚开始我用 `var` 用得过头，写出了这种代码：
```java
var i = 1; // 别人得猜这是 int 还是 long？
var data = getData(); // getData 返回啥？还得跳进去看源码
```
这让代码可读性灾难级下降。

**最佳实践：**
`var` 的正确打开方式，是拯救那些冗长到令人发指的泛型声明。

```java
// Before: 眼睛都看花了
Map<String, List<Map<String, User>>> userMap = new HashMap<>();

// After: 清爽利落，类型一目了然
var userMap = new HashMap<String, List<Map<String, User>>>();

// 流式操作中更是神器
var result = users.stream()
    .filter(u -> u.isActive())
    .collect(Collectors.toMap(User::getId, User::getName));
```
**记住：** 当右侧类型显而易见时，用 `var`；当右侧是个字面量或模糊方法调用时，请老实写下类型。

### 2.3 Switch 表达式与模式匹配：告别冗长的样板代码

**引入版本**：JDK 14 ~ 21 | **关键词**：`->`, `yield`, `instanceof`

以前处理多分支逻辑，简直是写 `if-else` 和 `break` 的体力活。更痛苦的是类型转换，每次都要先 `instanceof` 再强转，稍不留神就是 `ClassCastException`。

**高潮时刻（顿悟）：**
有一次我要解析一个复杂的 JSON 响应，里面嵌套了多种类型的对象。旧代码写了整整 50 行，充满了嵌套判断。当我尝试用 JDK 21 的模式匹配重写时，奇迹发生了：

```java
// Before: 冗长且危险
if (obj instanceof String) {
    String s = (String) obj;
    return s.length();
} else if (obj instanceof Integer) {
    Integer i = (Integer) obj;
    return i;
} else {
    return 0;
}

// After: JDK 21 模式匹配 + Switch 表达式
return switch (obj) {
    case String s -> s.length();
    case Integer i -> i;
    case null -> 0;
    default -> throw new IllegalStateException("Unexpected type");
};
```
**那一瞬间，我感觉代码不再是砖块堆砌的墙，而是流动的溪水。** 编译器帮我做了类型检查，我再也不用担心转换错误，逻辑聚焦得纯粹而锋利。

### 2.4 文本块 (Text Blocks)：优雅处理多行字符串

**引入版本**：JDK 15 | **关键词**：`"""`, 原生格式

谁没在 Java 里写过这样的 SQL？
```java
String sql = "SELECT * FROM users\n" +
             "WHERE id = " + userId + "\n" +
             "AND status = 'ACTIVE'";
```
满屏的 `+` 号和 `\n`，看着就头疼，稍微改个缩进还要重新数空格。

**解脱时刻：**
文本块（Text Blocks）用三个双引号 `"""` 解放了我们。

```java
String sql = """
    SELECT * FROM users
    WHERE id = %d
    AND status = 'ACTIVE'
    """.formatted(userId);
```
**反转来了：** 它不仅保留了格式，还能自动处理缩进！你在代码里怎么排版，字符串里就是什么样。写 JSON、XML、正则表达式时，这种“所见即所得”的快乐，谁用谁知道。

### 2.5 Record 类：为不可变数据而生

**引入版本**：JDK 16 | **关键词**：`record`, 不可变，POJO 瘦身

如果你曾经为一个简单的 DTO 写过构造函数、Getter、`equals`、`hashCode` 和 `toString`，那你一定体会过什么是“生命浪费”。一个几十行的类，有效代码可能只有两个字段。

**我踩过的坑：**
有一次修改 POJO 的 `equals` 方法漏了一个字段，导致缓存命中失败，排查了两天。

**救星登场：**
`record` 专为纯数据载体而生。

```java
// 一行搞定！自动生成所有样板代码
public record User(String name, int age, String email) {}

// 使用
var user = new User("Alice", 25, "alice@example.com");
System.out.println(user.name()); // 注意：访问器方法没有 get 前缀
```
**惊喜彩蛋：** 你还可以写“紧凑构造函数”来做校验，而不必重复定义字段。
```java
public record User(String name, int age) {
    public User {
        if (age < 0) throw new IllegalArgumentException("Age cannot be negative");
    }
}
```
代码行数减少 80%，语义却清晰了 200%。

### 2.6 虚拟线程 (Virtual Threads)：并发编程的范式转移

**引入版本**：JDK 21 (正式) | **关键词**：Project Loom, 百万并发

这是本文的**绝对高潮**。

长期以来，Java 的并发模型依赖于操作系统内核线程。创建一个线程开销巨大，所以我们在高并发场景下被迫使用复杂的 Reactor 模式（如 Netty），或者陷入 `CompletableFuture` 的回调地狱。代码写得像天书，调试起来像破案。

**险象环生的 Debug：**
我曾在一个高并发网关项目中，因为线程池耗尽，导致整个系统雪崩。为了优化，我引入了响应式编程，结果代码复杂度飙升，团队成员怨声载道。

**颠覆性发现：**
JDK 21 的虚拟线程（Virtual Threads）彻底改变了游戏规则。它们是用户态线程，由 JVM 调度，创建成本极低（纳秒级），内存占用极小。

```java
// 以前：不敢随便 new Thread，得小心翼翼管理线程池
ExecutorService executor = Executors.newFixedThreadPool(200);

// 现在：直接启动百万级虚拟线程，风格回归同步！
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1_000_000; i++) {
        executor.submit(() -> {
            // 这里的 I/O 阻塞不会卡住内核线程！
            // 就像写单线程代码一样简单
            var response = httpClient.send(request); 
            process(response);
        });
    }
}
```
**顿悟时刻：**
当我第一次看到监控图上，百万个任务在几个内核线程上流畅切换，而 CPU 占用率依然平稳时，我震惊了。**我们终于可以用同步的代码风格，写出异步的性能。** 这不是优化，这是重生。Go 语言的 Goroutine 曾让我们羡慕不已，现在，Java 做到了，而且更优雅。

### 2.7 字符串模板 (String Templates)：更安全高效的动态构建

**引入版本**：JDK 21 (预览) | **关键词**：`${}`, 处理器

虽然还在预览，但这绝对是未来的方向。以前的字符串拼接要么不安全（SQL 注入风险），要么不灵活（`String.format` 类型不安全）。

```java
// 未来展望 (预览特性)
String query = STR."SELECT * FROM users WHERE name = \{userName}";
```
更强大的是**自定义处理器**，可以在插值前自动进行转义或验证，从根源上杜绝注入攻击。这不仅仅是语法糖，这是安全防线的前移。

## 三、加分项：其他值得关注的底层革新

除了上述七大神器，还有一些“隐形冠军”在底层默默发力：

*   **ZGC 垃圾收集器**：从实验走向默认。它能将停顿时间控制在 1ms 以内，哪怕堆内存高达 TB 级。对于延迟敏感的微服务，它是救命稻草。
*   **外部函数与内存 API (Foreign Function & Memory API)**：终于，我们可以安全地告别危险的 `JNI` 和 `Unsafe` 了。调用 C 库、管理堆外内存变得既高效又类型安全。
*   **Scoped Values**：专为虚拟线程设计。它解决了 `ThreadLocal` 在虚拟线程环境下的内存泄漏和性能问题，是新一代的数据共享方案。

## 四、实战指南：从 JDK 8 平滑升级的策略

我知道，看到这里你可能心动了，但看着手里那堆跑在 JDK 8 上的老旧代码，心里又开始打鼓：“升级会不会炸？”

别怕，这是我总结的**平滑迁移路线图**：

1.  **评估与决策**：
    *   如果你的团队还在用 Spring Boot 2.x，请先升级到 Spring Boot 3.x（最低要求 JDK 17）。
    *   不要试图一步登天到 JDK 21。**建议路径：JDK 8 → JDK 17 → JDK 21**。JDK 17 是一个极其稳定的中间站。

2.  **扫描与修复**：
    *   使用官方工具 `jdeprscan` 扫描项目中使用的已废弃 API。
    *   解决模块化带来的非法反射访问问题（临时可用 `--add-opens`，但长远需重构代码）。

3.  **渐进式重构**：
    *   **第一阶段**：只改构建配置，确保能跑。
    *   **第二阶段**：引入语法糖。把 `var`、`Switch` 表达式、`Record` 用在新增代码中，逐步替换旧代码。
    *   **第三阶段**：架构级演进。当代码库足够现代后，再考虑引入虚拟线程重构高并发模块。

4.  **容器化优化**：
    *   利用 `jlink` 或 JIB 插件，构建精简的 Docker 镜像。你会发现，你的镜像体积可能缩小一半以上。

## 六、常见问题解答 (FAQ)

**Q1: 生产环境直接用 JDK 21 稳吗？**
**A:** 非常稳。JDK 21 是 LTS（长期支持）版本，经过了充分的测试。主流框架（Spring Boot 3.2+）都已完美支持。如果是全新项目，强烈推荐直接上 21；如果是老项目，建议先升到 17 过渡。

**Q2: 虚拟线程会完全取代平台线程（内核线程）吗？**
**A:** 不会完全取代，但会成为默认首选。对于计算密集型任务（如大量数学运算），平台线程仍有优势；但对于绝大多数 I/O 密集型应用（Web 服务、数据库交互），虚拟线程是绝对的主宰。

**Q3: 升级后，旧的第三方库不兼容怎么办？**
**A:** 大部分主流库都已适配。如果遇到使用了内部反射的老库，可以先用 `--illegal-access=permit` (JDK 9-16) 或 `--add-opens` (JDK 17+) 临时绕过，同时联系库作者更新，或者寻找替代品。模块化虽然痛，但它逼着生态变得更健康。

**Q4: `record` 类可以继承其他类吗？**
**A:** 不可以。`record` 隐式继承自 `java.lang.Record`，且是 `final` 的。它的设计初衷就是作为不可变的数据载体，而不是复杂的业务逻辑容器。如果需要继承，请使用普通类。
