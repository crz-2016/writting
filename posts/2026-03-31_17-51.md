# `ArrayList` 高并发事故复盘与架构决策

凌晨三点核心交易链路雪崩，监控显示 CPU 击穿 100%。排查确认无外部攻击或数据库死锁，根因为高并发下 `ArrayList` 遍历触发大规模扩容。多线程同时检测到容量不足，竞相执行 `System.arraycopy` 进行内存拷贝，导致年轻代 GC 压力激增，频繁触发 Stop-The-World，服务僵死。

## 源码机制：懒加载、位运算与 Fail-Fast

JDK 8+ 中 `new ArrayList()` 并非立即分配默认 10 个空间，而是初始化长度为 0 的 `DEFAULTCAPACITY_EMPTY_ELEMENTDATA` 数组。**仅在首次调用 `add` 时才分配默认容量**。此懒加载策略在单线程节省内存，但在高并发写入空列表时，会导致多线程同时触发扩容逻辑，引发锁竞争与重复拷贝。

扩容算法采用位运算实现 1.5 倍增长：`int newCapacity = oldCapacity + (oldCapacity >> 1)`。**使用右移替代除法是为了消除指令周期开销，1.5 倍系数是内存占用率与扩容频率之间的数学最优解**，而非随意设定。

结构修改通过 `modCount` 变量追踪。迭代器启动时记录预期 `modCount`，每次 `next()` 调用校验该值。**Fail-Fast 机制并非 Bug，而是防止在数据结构变更期间产生脏读数据的强制性一致性约束**。

## 性能权衡：时间复杂度与硬件架构的博弈

`ArrayList` 的 `get(i)` 为 O(1)，基于内存地址偏移；但中间插入/删除需移动后续所有元素，复杂度为 O(n)。**在现代 CPU 架构下，由于 `ArrayList` 具备优秀的缓存局部性（Cache Locality），其内存拷贝速度往往优于 `LinkedList` 的指针跳转**，后者因节点堆内存分散导致大量 Cache Miss。

自动扩容伴随旧数组丢弃与新数组分配，海量数据下会产生内存碎片并加重 GC 负担。**生产环境必须显式指定 `initialCapacity`，依赖默认扩容机制等同于主动引入性能抖动**。

`Collections.synchronizedList` 仅保证单操作原子性，复合操作（如检查后添加）仍存在竞态条件。**`CopyOnWriteArrayList` 通过写时复制整个数组实现读无锁，这是以 O(n) 的写开销换取读操作零锁竞争的极端策略，仅适用于读多写极少场景**。

## 架构决策模型

`ArrayList` 的设计预设了“读多于写、追加多于插入、单线程多于多线程”的场景。**数据结构选型本质是业务场景与底层资源（CPU 缓存、GC 压力、内存带宽）的匹配过程，不存在通用最优解**。

实例化前必须明确三点：
1. 数据量级是否已知且可预估初始容量？
2. 是否存在高频中间插入操作？
3. 是否涉及多线程共享及读写比例？

**高级工程能力体现为在实例化瞬间即可量化该选择带来的具体性能账单（GC 频率、CPU 指令数、缓存命中率），而非单纯记忆 API 用法**。
